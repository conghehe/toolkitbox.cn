关于 JWT (JSON Web Token)

  
JWT（JSON Web Token）是一种开放标准 (RFC 7519)，定义了一种紧凑且自包含的方式，用于在各方之间以 JSON 对象安全传输信息。JWT 广泛应用于现代 Web 应用的身份认证和信息交换中。


  
JWT 结构

  
JWT 由三部分组成，用点号 (.) 分隔：

  
    
    
  • Header（头部）：包含令牌类型 (JWT) 和签名算法 (如 HS256、RS256)
    • 
    
  • Payload（负载）：包含声明 (Claims)，如用户ID、过期时间等
    • 
    
  • Signature（签名）：用于验证消息在传输过程中未被篡改
    • 
  


  
常见声明 (Claims)

  

    
声明全称说明

    
issIssuer签发者

    
subSubject主题（通常是用户ID）

    
audAudience接收方

    
expExpiration Time过期时间（Unix时间戳）

    
nbfNot Before生效时间

    
iatIssued At签发时间

    
jtiJWT ID唯一标识符

  


  
常见签名算法

  

    
算法类型安全性

    
HS256HMAC + SHA-256对称密钥，适合单服务

    
HS384HMAC + SHA-384对称密钥，安全性更高

    
HS512HMAC + SHA-512对称密钥，最高安全性

    
RS256RSA + SHA-256非对称，适合微服务

    
ES256ECDSA + SHA-256非对称，性能更优

    
none无签名⚠️ 不安全，禁止生产使用

  


  
使用场景

  
    
    
  • 身份认证：用户登录后，服务器生成JWT返回客户端，后续请求携带JWT验证身份
    • 
    
  • 信息交换：在微服务之间安全传递信息，JWT可使用公钥/私钥签名确保来源可信
    • 
    
  • 单点登录 (SSO)：在多个系统之间共享认证状态
    • 
    
  • API 授权：OAuth 2.0 中使用 JWT 作为 Access Token
    • 
  


  
安全注意事项

  
    
    
  • 永远不要在前端存储敏感信息在 JWT Payload 中（Payload 仅 Base64 编码，非加密）
    • 
    
  • 建议设置合理的过期时间 (exp)，推荐 15 分钟到 24 小时
    • 
    
  • 使用 HTTPS 传输 JWT，防止中间人攻击
    • 
    
  • 避免使用 "none" 算法，确保服务器强制验证签名算法
    • 
    
  • 不要在 URL 参数中传递 JWT（会被记录在服务器日志中）
    • 
    
  • 考虑使用 Refresh Token 机制，缩短 Access Token 有效期
    • 
  


  
常见问题

  
Q: JWT 安全吗？ JWT 的 Payload 是 Base64 编码而非加密，任何人都可以解码查看。安全性在于签名验证——只要签名正确，就可以信任数据未被篡改。

  
Q: JWT 和 Session 哪个更好？ JWT 适合分布式/微服务架构，Session 适合单体应用。JWT 无状态，Session 有状态。选择取决于你的架构需求。

  
Q: JWT 过期了怎么办？ 通常使用 Refresh Token 机制。Access Token 短期有效（如15分钟），过期后用 Refresh Token 获取新的 Access Token。

  
Q: 本工具安全吗？ 本工具所有解析完全在浏览器本地执行，JWT 数据不会上传到任何服务器。你可以放心使用，甚至离线使用。
